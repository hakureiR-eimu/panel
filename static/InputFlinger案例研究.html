项目背景

InputFlinger 在 Android 中的角色

InputFlinger 是 Android 系统的 Native 输入事件处理管线，负责从内核设备节点读取原始输入事件（EventHub），经过处理和转换（InputReader），最终分发给目标窗口（InputDispatcher）。整个流程通过 JNI 桥接层（InputManagerService.cpp）与 Java 层的 InputManagerService 连接。

组件 职责 仓库归属
EventHub 从 /dev/input/ 设备节点读取原始事件，管理设备热插拔 2014 前在 base，后在 native
InputReader 将原始事件转换为高级输入事件（Motion/Key），管理输入设备配置 同上
InputDispatcher 将高级事件分发到目标窗口，管理 ANR 超时、焦点切换 同上
InputManagerService (JNI) Java ↔ Native 桥接，注册 JNI 方法，传递策略配置 始终在 base

综合总结

演化叙事：InputFlinger 在 frameworks/base 中经历了"构建 → 完善 → 仓库内迁移 → 跨仓库迁出"的完整生命周期。JNI 桥接层 InputManagerService.cpp 作为唯一留守文件，持续演化 13 年，是 Java 框架层与 Native 输入系统之间永久的桥梁。

关键发现

  1. 两次迁移，两种性质 层 2

    2013 年是仓库内目录重组（FILE_DIR_CHANGE，986 次），代码不变仅路径变。2014 年是跨仓库迁出（ADD+DELETE，1,649 次），标识符重建。

  2. InputReader 是最活跃的文件 层 2

    无论是 ADD（421）、DELETE（637）还是参数变更，InputReader 始终排在首位，反映了输入处理逻辑的复杂性和持续演进。

  3. 接口快速收敛 层 2

    重命名事件从 2011 年的 38 次降到 2012 年的 10 次，说明核心接口在一年内趋于稳定。2020 年的 19 次重命名对应 Android 大版本 API 变更。

  4. JNI 层的生命力 层 2

    InputManagerService.cpp 持续活跃 13 年，2024 年仍有 56 个 commit、375 次事件，说明 Android 框架与 Native 输入系统的接口仍在持续演进。

局限与 Caveats

Co-change 分析不可用：本次扫描的 co-change 预计算因磁盘空间不足而失败，无法进行 dim4（已消除坏味道）和 dim5（残留坏味道）分析。
  • 数据覆盖不完整：InputReader/InputDispatcher/EventHub 仅有 2011-2014 年的数据，2014 年后的演化发生在 platform_frameworks_native 中，不在本数据库覆盖范围内。
  • 无 unified_id 映射表：本次扫描的子库中没有 identifier_unified_id_mapping 表，无法区分同一函数的多条独立 lineage 与单条 lineage 的演化。
  • DELETE 保守：v2 的 DELETE 识别偏保守，部分迁出可能记录为路径变更而非删除。

数据来源与复现

SQL 查询：年度活跃度直方图 
SELECT YEAR(c.author_time) AS yr,
       COUNT(DISTINCT ic.commit_id) AS commits_touched,
       COUNT(*) AS event_count,
       SUM(CASE WHEN ctd.name = 'ADD' THEN 1 ELSE 0 END) AS adds,
       SUM(CASE WHEN ctd.name = 'DELETE' THEN 1 ELSE 0 END) AS deletes,
       SUM(CASE WHEN ctd.name = 'FILE_DIR_CHANGE' THEN 1 ELSE 0 END) AS dir_changes,
       SUM(CASE WHEN ctd.name = 'IDENTIFIER_NAME_CHANGE' THEN 1 ELSE 0 END) AS renames
FROM identifier_change ic
JOIN identifier i ON i.hashed_uid = ic.identifier_id
JOIN change_type_dict ctd ON ctd.id = ic.change_type_id
JOIN commit c ON c.id = ic.commit_id
WHERE i.path IN (
    'services/input/InputReader.cpp',
    'libs/input/InputReader.cpp',
    'services/input/InputDispatcher.cpp',
    'libs/input/InputDispatcher.cpp',
    'services/input/EventHub.cpp',
    'libs/input/EventHub.cpp',
    'services/core/jni/com_android_server_input_InputManagerService.cpp',
    'services/jni/com_android_server_input_InputManagerService.cpp'
)
GROUP BY yr ORDER BY yr;
SQL 查询：职责迁移（CROSS_FILE_MOVE / FILE_DIR_CHANGE） 
SELECT YEAR(c.author_time) AS yr,
       src.path AS src_path, dst.path AS dst_path,
       i.name AS func_name, t.name AS relation_type, COUNT(*) AS cnt
FROM identifier_relation ir
JOIN identifier src ON src.hashed_uid = ir.src_identifier_id
JOIN identifier dst ON dst.hashed_uid = ir.dst_identifier_id
JOIN change_type_dict t ON t.id = ir.change_type_id
JOIN commit c ON c.id = ir.commit_id
JOIN identifier i ON i.hashed_uid = ir.src_identifier_id
WHERE src.path LIKE '%/input/%'
  AND t.name IN ('CROSS_FILE_MOVE', 'FILE_DIR_CHANGE')
GROUP BY yr, src.path, dst.path, i.name, t.name
ORDER BY yr, cnt DESC;
