3. 阶段划分 · 跨仓库视角

2011 初创期 base — InputReader.cpp 贡献 999 次事件，从零构建输入系统。EventHub 从 InputDispatcher 抽取 InputWindow 职责（4 个 CROSS_FILE_MOVE）。
2012 功能完善 base — InputManagerService.cpp（JNI）开始活跃。Java 层 InputManagerService.java 建立初始 API 体系。
2013 模块化迁移 base — 622 次 FILE_DIR_CHANGE，services/input/ → libs/input/，为独立仓库做准备。
2014 跨仓库迁出 base→native — 1,779 次 base 事件（含大量 DELETE）+ native 开始有 2,314 次事件。InputFlinger 整体迁出。
2015-17 独立演化 native — 年均 ~250 事件，引入新能力。
2018 基类提取 native — InputReaderBase.cpp 抽取。InputWindow.cpp → libs/input/ 共享库。
2019 大拆分重构 native — 4,977 事件！2,530 次 FILE_DIR_CHANGE + 875 次 CROSS_FILE_MOVE。InputReader/InputDispatcher 拆分为 reader/ 和 dispatcher/ 子目录。
2020-25 能力扩展期 native — 年均 4,000+ 事件。新增 InputClassifier、MotionPredictor、PointerChoreographer、InputProcessor 等模块。

7. 总结与关键发现

三次架构转折
  1. 2014 跨仓库迁出：从 frameworks/base 独立到 frameworks/native，Android 输入系统从"框架附属"变为"独立子系统"
  2. 2019 大拆分：InputReader 和 InputDispatcher 从单体巨石文件拆分为 reader/ 和 dispatcher/ 子目录，每个 mapper 类型独立成文件
  3. 2020-2024 能力扩展：在模块化架构基础上持续引入新能力——ML 分类、运动预测、触控板手势、Perfetto 追踪等
持续关注点
  • InputDispatcher.cpp 仍是最大文件：7,335 行，核心分发逻辑高度集中，可能面临下一轮拆分压力
  • 新能力引入节奏加速：2020 年后年均新增标识符 600-950 个，远超 2015-2017 年的 50-80 个，需警惕新"巨石文件"形成

-- 年度活跃度
SELECT YEAR(c.commit_time) as yr, COUNT(DISTINCT ic.commit_id) as commits,
  COUNT(*) as events, SUM(CASE WHEN ct.id=1 THEN 1 ELSE 0 END) as adds,
  SUM(CASE WHEN ct.id=2 THEN 1 ELSE 0 END) as dels
FROM identifier_change ic
JOIN identifier i ON ic.identifier_id = i.hashed_uid
JOIN commit c ON ic.commit_id = c.id
JOIN change_type_dict ct ON ic.change_type_id = ct.id
WHERE i.path LIKE 'services/inputflinger/%' AND ct.id NOT IN (19,28)
GROUP BY yr ORDER BY yr;

-- CROSS_FILE_MOVE 详情
SELECT i.name, i.path as src, i2.path as dst, YEAR(c.commit_time) as yr
FROM identifier_relation ir
JOIN identifier i ON ir.src_identifier_id = i.hashed_uid
JOIN identifier i2 ON ir.dst_identifier_id = i2.hashed_uid
JOIN commit c ON ir.commit_id = c.id
WHERE ir.change_type_id = 21
  AND i.path LIKE 'services/inputflinger/%'
ORDER BY yr, i.name;
